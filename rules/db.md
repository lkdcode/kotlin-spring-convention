# rules/db — DDL 컨벤션

---

## 테이블 네이밍

**도메인이 먼저, 접두사(`mst_`, `his_`) 금지.**

```
{domain}                      ← 주 엔티티 (마스터성)
{domain}_{entity}             ← 하위 엔티티 / 관계 테이블
{domain}_{entity}_history     ← 이력성
{domain}_{entity}_log         ← 로그성
{domain}_{entity}_snapshot    ← 스냅샷
```

### 특수 prefix (도메인과 분리)

도메인 테이블이 아닌 시스템 목적 테이블은 prefix 를 사용해 도메인과 명확히 분리.

```
batch_{name}      ← 배치 처리 관련
agg_{name}        ← 집계/통계 데이터
```

```sql
-- 예시
batch_job_execution      ← 배치 실행 이력
agg_daily_user_stats     ← 일별 사용자 통계
agg_monthly_sales        ← 월별 매출 집계
```

### 예시

| ❌ 기존 방식 | ✅ 도메인 우선 |
|---|---|
| `mst_user` | `users` |
| `mst_user_group` | `users_groups` |
| `his_login` | `users_login_history` |
| `his_display` | `users_display_history` |
| `mst_material` | `materials` |
| `his_material_receipt` | `materials_receipt_history` |
| `mst_display_role` | `displays_roles` |
| `his_project_part` | `projects_part_history` |

### 규칙

- 소문자 snake_case
- 복수형 — `user` → `users`, `order` → `orders`
- 도메인명이 항상 첫 번째 — `{domain}_{설명}`
- 마스터성: 도메인명만 (`users`, `products`)
- 이력/로그성: `_history`, `_log`, `_snapshot` 접미사
- 매핑 테이블: `{domain_a}_{domain_b}` (알파벳 순서)

## 컬럼 네이밍

소문자 snake_case.

```sql
-- 기본 컬럼
id              ← PK
{domain}_id     ← FK (참조 테이블의 도메인명 + _id)
name            ← 이름
code            ← 코드 (유니크한 식별 문자열)
status          ← 상태 (ENUM 또는 VARCHAR)
type            ← 유형

-- 시간
created_at
updated_at
deleted_at
started_at      ← 특정 시작 시간
ended_at        ← 특정 종료 시간

-- 불리언
is_deleted      ← 소프트 삭제
is_{상태}       ← 상태 플래그 (is_active, is_locked)

-- BaseEntity 공통
created_at, updated_at, created_by, updated_by
is_deleted, deleted_at
```

### 규칙

- FK 컬럼: `{참조도메인}_id` — 예: `user_id`, `product_id`
- 불리언 컬럼: `is_` prefix — 예: `is_deleted`, `is_active`
- 시간 컬럼: `_at` suffix — 예: `created_at`, `started_at`
- 약어 사용 금지 — `usr_id` ❌, `user_id` ✅

## ID 생성 방식

두 가지 방식 중 프로젝트에서 선택.

| 방식 | 특징 | 적합한 경우 |
|---|---|---|
| **Sequence (Auto Increment)** | 순차 증가, 정수 | 단일 DB, 단순 CRUD |
| **UUIDv7** | 시간 정렬 가능 UUID | 분산 환경, 노출 방지 필요 |

### Sequence

```sql
-- MySQL
id  BIGINT UNSIGNED  NOT NULL  AUTO_INCREMENT PRIMARY KEY

-- PostgreSQL
id  BIGINT  GENERATED BY DEFAULT AS IDENTITY  PRIMARY KEY
```

### UUIDv7

시간 기반 정렬이 가능한 UUID. 기존 UUIDv4 보다 인덱스 성능 우수.

```sql
-- MySQL
id  VARCHAR(36)  NOT NULL  PRIMARY KEY  COMMENT '식별 인덱스 (UUIDv7)'

-- PostgreSQL
id  UUID  NOT NULL  DEFAULT gen_random_uuid()  PRIMARY KEY
```

## 외래키 (FK)

**필요한 FK 는 반드시 추가.** 참조 무결성을 DB 레벨에서 보장.

```sql
-- MySQL
CONSTRAINT fk_users_groups_users
    FOREIGN KEY (user_id) REFERENCES users (id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE,

-- PostgreSQL
CONSTRAINT fk_users_groups_users
    FOREIGN KEY (user_id) REFERENCES users (id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE
```

### FK 네이밍

```
fk_{현재테이블}_{참조테이블}

fk_users_groups_users        ← users_groups.user_id → users.id
fk_orders_products           ← orders.product_id → products.id
```

### 규칙

- FK 는 원칙적으로 추가 — 참조 무결성 DB 레벨 보장
- `ON DELETE RESTRICT` 기본 — 참조 데이터 삭제 방지
- soft delete 테이블은 `ON DELETE RESTRICT` 로 충분 (물리 삭제 없음)

## 인덱스 네이밍

```
idx_{table}_{column(s)}

idx_users_email
idx_users_login_history_user_id
idx_orders_created_at
idx_orders_user_id_status       ← 복합 인덱스
```

### 유니크 인덱스

```
uq_{table}_{column}

uq_users_email
uq_materials_code
```

## 테이블 구조 순서

```sql
CREATE TABLE users
(
    -- 1. PK
    id,

    -- 2. 비즈니스 컬럼 (도메인 핵심 데이터)
    name,
    email,
    status,

    -- 3. FK 컬럼
    user_group_id,

    -- 4. BaseEntity (생성/수정/삭제)
    created_at,
    updated_at,
    created_by,
    updated_by,
    is_deleted,
    deleted_at,
);
```

| 불리언 | `is_` prefix |
| 시간 | `_at` suffix |

| 유니크 인덱스 | `uq_{table}_{column}` |
